# Технические решения и trade-off'ы

Коротко — почему сделано именно так, а не иначе. Это то, что стоит проговорить на собеседовании,
если спросят "а почему вот так".

### Один основной warehouse (ClickHouse), а не два
Исходная версия проекта дублировала каждую трансформацию под ClickHouse и Greenplum одновременно.
Это удваивало объём поддерживаемого кода почти без дополнительной пользы: обе версии решали одну
и ту же задачу. Решение — сделать ClickHouse основным путём (dbt, Airflow, CI), а Greenplum-скрипты
оставить как референс в `sql/legacy_greenplum/`, чтобы показать, что дизайн данных warehouse-agnostic,
не тратя время на поддержку двух живых пайплайнов ради демо-проекта.

### dbt вместо голых SQL-скриптов, запускаемых вручную
В скелете трансформации были обычными `.sql`-файлами, которые нужно было запускать руками
(`clickhouse-client --multiquery < ...`). Это не масштабируется: нет тестов, нет линейного DAG
зависимостей между моделями, нет documentation-as-code. dbt даёт всё это "из коробки" и является
одним из самых частых требований в вакансиях Data Engineer.

### `materialized='table'` (full refresh), а не incremental, для `seller_balance_daily`
`running_balance` — кумулятивная величина по всей истории продавца. Наивный incremental-пересчёт
только "новых" дней даёт неверный накопленный баланс на границе инкремента (нужно откуда-то брать
opening balance на начало окна). При объёме данных этого проекта (десятки продавцов, месяцы истории)
полный пересчёт в ClickHouse занимает миллисекунды — платить корректностью за экономию на
вычислениях тут бессмысленно. При росте на порядки: инкрементальный пересчёт с отдельно хранимым
opening balance на начало каждого нового окна.

### Дедупликация — в staging-слое (dbt), а не только в DQ
Источники в реальной жизни работают по принципу "at-least-once" (ретраи, повторная доставка) —
дубликаты это норма, а не аномалия. `stg_events.sql` дедуплицирует по `event_id` через
`ROW_NUMBER()`, поэтому марты всегда строятся на чистых данных. DQ отдельно всё равно проверяет
дубликаты в raw-слое — это два независимых уровня защиты, а не дублирование одной и той же проверки.

### Два независимых уровня DQ, а не один
1. `dq/checks.py` — построчные проверки Python сразу после загрузки в raw (до dbt): null-поля,
   дубликаты, outliers по сумме, доля "грязных" строк выше порога. Падает Airflow-таск.
2. dbt schema-тесты + singular-тест на сверку `net_change = inflow - outflow` — проверяют уже
   трансформированную витрину. Падает `dbt test`.

Так пайплайн не запускает дорогие трансформации на заведомо плохих данных (проверка №1 отсекает их
раньше), и одновременно гарантирует арифметическую согласованность результата (проверка №2).

### Идемпотентная загрузка (delete+insert по дате), а не append
`raw.events` в ClickHouse — обычный `MergeTree`, без встроенного upsert. Повторный запуск DAG за уже
обработанный день не должен плодить дубли. Решение — `ALTER TABLE ... DELETE WHERE toDate(event_ts) = ds`
перед вставкой (см. `pipeline/loader.py::load_day`). Это стандартный ClickHouse-паттерн для
батчевой перезаливки партиции.

### CI не поднимает полноценный Airflow
`dag-syntax-check` в CI ограничивается `python -m py_compile` для DAG-файла — поднятие метабазы
Airflow в GitHub Actions ради синтаксической проверки одного линейного DAG не окупает время сборки.
Реальный прогон DAG проверяется вручную через `make up` и веб-интерфейс Airflow
(http://localhost:8080). CI вместо этого гоняет то, что действительно можно и нужно проверять
автоматически: юнит-тесты DQ-модуля, лint, и dbt run/test против настоящего ClickHouse
(service container).

### Что бы изменилось при кратном росте нагрузки
- Near-real-time вместо batch: Kafka/Debezium CDC → raw-слой вместо ежедневной перезаливки CSV.
- `seller_balance_daily` — на incremental с отдельно хранимым opening balance.
- Алертинг: `_notify_failure` — сейчас лог-стаб, в проде — вебхук в Slack/PagerDuty с контекстом
  (dag_id, task_id, ссылка на логи).
- DQ-пороги и метрики — в Prometheus/Grafana, а не только в логах Airflow.
- Feature store для ML (например, витрина последних N дней по продавцу) — если баланс используется
  как фича в моделях риска/скоринга.
